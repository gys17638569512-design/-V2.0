1.2 Codex与Trae搭配使用规则
核心原则：两个工具读同一份AGENTS.md规范文件，产出的代码风格和结构完全一致。切换工具时，新工具第一件事必须读AGENTS.md和PROGRESS.md，然后才能开始工作。

场景 | 使用工具 | 原因
后端API开发 | Codex优先 | Codex对Python/FastAPI支持更好，可直接读GitHub仓库
Vue3前端组件 | Trae CN优先 | Trae对Vue3/Element Plus框架适配更好
Codex配额不足 | 切换Trae CN | 读AGENTS.md后无缝继续，不影响代码质量
排查Bug | Codex优先 | Codex可SSH连服务器查看实时日志
数据库迁移 | Codex | Codex可以在Docker容器内执行alembic命令
代码审查 | Claude.ai | 把代码描述给我，我帮你检查问题

1.3 ChatGPT Plus配额管理策略
ChatGPT Plus账号Codex有每日使用配额限制。必须合理分配，避免一天配额用完后无法继续开发。

大任务必须拆小：每次只让Codex做一个函数或一个页面，不要一次生成整个模块
优先级排序：复杂逻辑（数据隔离/状态机/权限控制）用Codex，简单页面用Trae
配额预警：每天下午3点前完成当天重要任务，预留配额给下午可能的调试工作
切换标准：Codex提示"已达到使用限制"时立即切换Trae，任务描述复制过去继续
次日重置：配额通常每24小时重置，重置后优先完成昨天未完成的复杂任务

# DMMS 数字化维保系统 · AI开发规范（AGENTS.md） # 所有AI工具必须在开始任何工作前读取本文件 # 版本：v1.0 · 湖南智造机械科技有限公司  ## 项目基本信息 - 系统名称：数字化维保系统（DMMS） - 公司：湖南智造机械科技有限公司 - 仓库结构：Monorepo（所有端在同一仓库） - 部署环境：Ubuntu 22.04 + Docker + Docker Compose  ## 技术栈（锁定，不可更改） 后端：Python 3.10+ / FastAPI / SQLAlchemy 2.x / MySQL 8.0 PC前端：Vue3 3.4+ / Element Plus / Pinia / TypeScript / Axios 小程序：uni-app（非原生微信小程序） 缓存：Redis（通知/Session） 文件存储：阿里云OSS 容器：Docker + Docker Compose  ## 绝对禁止事项 ❌ 禁止在代码里硬编码任何密码、密钥、手机号 ❌ 禁止跳过Alembic直接修改数据库结构 ❌ 禁止在Router层直接操作数据库（必须经过Service层） ❌ 禁止在前端代码里硬编码下拉选项值（必须从接口取） ❌ 禁止一次生成超过1个模块的代码（必须逐模块开发） ❌ 禁止跳过pytest直接进入下一个模块 ❌ 禁止修改已有的数据库迁移文件  ## 后端分层架构（严格遵守） 请求 → routers/（参数验证+权限检查）       → services/（业务逻辑+数据隔离）       → repositories/（数据库CRUD）       → models/（ORM表结构）  ## 数据隔离规则（必须在Service层实现） MANAGER角色：查询时追加 .filter(Customer.manager_id == current_user.id) TECH角色：查询时追加 .filter(WorkOrder.engineer_id == current_user.id) CLIENT角色：只能看自己企业的数据 ADMIN角色：无限制，看本发展商所有数据  ## 统一响应格式 成功：{"code": 0, "msg": "ok", "data": {...}} 失败：{"code": 错误码, "msg": "错误描述", "data": null} 错误码：400参数错误/401未授权/403无权限/404不存在/500服务器错误  ## 命名规范 Python文件/函数/变量：snake_case Python类：PascalCase Vue组件文件：PascalCase.vue Vue变量：camelCase 数据库表名：snake_case复数（work_orders, customers） API路径：kebab-case复数（/api/v1/work-orders） Git提交格式：feat(模块): 描述 / fix(模块): 描述 / refactor(模块): 描述  ## 环境变量规则 所有配置从.env文件读取，不硬编码在代码里 后端：DATABASE_URL / SECRET_KEY / OSS_* / SMS_* / WECHAT_* 前端：VITE_API_BASE_URL（开发时指向本地/生产时指向服务器）  ## 每个模块完成后必须执行 1. 后端：运行 pytest tests/ -v 确认全绿 2. 前端：运行 npm run build 确认无错误 3. Git提交：git add . && git commit -m "feat(模块名): 完成XX功能" 4. 更新PROGRESS.md：把当前模块标记为✅已完成

2.4 API_CONTRACT.md 接口契约（关键片段）
接口契约是防止前后端AI格式不一致的核心文件。所有接口的输入输出格式在这里定义，前端和后端都照着做，不允许自己发明格式。

# API接口契约（API_CONTRACT.md）  ## 基础规范 Base URL: /api/v1 认证方式: Authorization: Bearer {JWT_TOKEN} 内容类型: Content-Type: application/json  ## 统一响应格式 成功响应： {   "code": 0,   "msg": "ok",   "data": {...} 或 [...] }  分页响应： {   "code": 0,   "msg": "ok",   "data": {     "items": [...],     "total": 100,     "page": 1,     "page_size": 20   } }  错误响应： {   "code": 401,   "msg": "未授权，请重新登录",   "data": null }  ## 分页参数（统一） GET /api/v1/work-orders?page=1&page_size=20&status=PENDING  ## 时间格式 所有时间字段使用ISO 8601格式：2026-04-19T14:30:00+08:00  ## 核心接口定义示例  ### 登录 POST /api/v1/auth/login 请求：{"username": "string", "password": "string"} 响应data：{   "access_token": "string",   "token_type": "bearer",   "expires_in": 86400,   "user": {"id":1,"username":"admin","role":"ADMIN","name":"张三"} }  ### 工单列表 GET /api/v1/work-orders 响应data.items[0]：{   "id": 1,   "order_no": "WO-2026-000001",   "customer_name": "宝武钢铁",   "equipment_name": "QD50T桥式起重机A2",   "engineer_name": "张伟",   "status": "PENDING",   "priority": "NORMAL",   "planned_date": "2026-04-20",   "is_overdue": false,   "overdue_days": 0,   "created_at": "2026-04-19T10:00:00+08:00" }  ### 工单状态枚举 PENDING/ASSIGNED/ACCEPTED/IN_PROGRESS/PENDING_SIGN/COMPLETED/CANCELLED/REVIEWED  ### 工单优先级枚举 NORMAL/URGENT  ### 角色枚举 SUPER_ADMIN/ADMIN/MANAGER/TECH/CLIENT/DEALER

name: Deploy to Production  on:   push:     branches: [main]  jobs:   deploy:     runs-on: ubuntu-latest     steps:       - name: SSH到服务器执行部署         uses: appleboy/ssh-action@master         with:           host: ${{ secrets.SERVER_HOST }}           username: ${{ secrets.SERVER_USER }}           key: ${{ secrets.SERVER_SSH_KEY }}           script: |             cd /opt/dmms             git pull origin main             docker-compose -f docker-compose.prod.yml build backend             docker-compose -f docker-compose.prod.yml up -d             docker-compose -f docker-compose.prod.yml exec backend               alembic upgrade head             echo "部署完成"

5.2 数据库迁移规范
禁止直接修改数据库！每次改动数据库结构（加字段/加表/改字段类型），必须用Alembic生成迁移文件，提交到Git。生产环境通过运行迁移文件来更新数据库结构。

# 正确的数据库变更流程：  1. 修改 models/ 下的ORM模型文件  2. 生成迁移文件    docker-compose exec backend alembic revision --autogenerate -m "描述"    例如：alembic revision --autogenerate -m "add_phone_to_user"  3. 检查生成的迁移文件（在migrations/versions/目录）    确认upgrade()和downgrade()函数内容正确  4. 执行迁移    docker-compose exec backend alembic upgrade head  5. 提交迁移文件到Git    git add migrations/    git commit -m "db: add phone field to user table"  # 禁止的操作： ❌ 直接在数据库工具里加字段 ❌ 修改已经执行过的迁移文件 ❌ 删除迁移文件

// src/utils/websocket.ts // WebSocket连接管理，含断线重连  class WSManager {   private ws: WebSocket | null = null   private reconnectDelay = 1000   private maxDelay = 30000    connect(token: string) {     const url = `${import.meta.env.VITE_WS_URL}/ws?token=${token}`     this.ws = new WebSocket(url)      this.ws.onopen = () => {       console.log("WebSocket已连接")       this.reconnectDelay = 1000  // 重置重连延迟     }      this.ws.onmessage = (event) => {       const msg = JSON.parse(event.data)       // 触发通知事件，通知铃铛更新       window.dispatchEvent(new CustomEvent("ws-notification", {detail: msg}))     }      this.ws.onclose = () => {       // 指数退避重连       setTimeout(() => {         this.reconnectDelay = Math.min(this.reconnectDelay * 2, this.maxDelay)         this.connect(token)       }, this.reconnectDelay)     }   } }  export const wsManager = new WSManager()

第十章  AI任务失败处理规范
10.1 任务失败场景处理
失败场景 | 处理方法 | 恢复步骤
AI生成代码后pytest报错 | 不要继续，先修复当前错误 | 把报错信息粘贴给AI，让它修复，直到pytest全绿
AI生成代码导致程序无法启动 | git stash保存现场，回滚到上次提交 | git stash / git checkout HEAD~1 / 重新给任务
Codex配额用完，任务未完成 | 切换Trae CN继续，读AGENTS.md | 复制任务描述到Trae，从上次完成的地方继续
AI理解错了需求，代码方向偏了 | 停止，不要继续，重新给任务描述 | git stash清理/重写任务描述/重新开始该模块
数据库迁移冲突 | 不要手动改迁移文件 | 让AI生成新迁移文件覆盖，或alembic downgrade回退
前后端接口格式不一致 | 对照API_CONTRACT.md检查 | 让前端AI或后端AI修改，对齐格式

10.2 代码回滚操作（让Codex执行）
# 场景1：最近几次提交都有问题，回到某个正常版本 git log --oneline -10           # 查看最近10次提交记录 git checkout abc1234            # 临时查看某个版本 git revert abc1234             # 撤销某次提交（保留历史）  # 场景2：当前改动还没提交，想丢弃回到上次提交 git checkout -- .              # 丢弃所有未提交改动  # 场景3：暂存当前改动，切换去做其他事 git stash                      # 暂存 git stash pop                  # 恢复暂存  # 场景4：错误合并了代码，需要撤销 git revert HEAD                # 撤销最后一次提交 git push origin develop        # 推送撤销记录
